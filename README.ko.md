# slide-maker

> **발표용 수준의 `.pptx` 덱을 만들고, 다시 디자인하고, 비평합니다** — 어떤 청중에게든, 어떤 언어로든, 템플릿이나 원본 자료가 있든 없든.

대부분의 AI 도구는 슬라이드를 텍스트 만들듯 만듭니다. 추측에 기대어 단번에 뽑아내고, 자기가 무엇을 만들어냈는지 한 번도 *들여다보지* 않습니다. **slide-maker는 그 대신 노련한 발표 디자이너처럼 일합니다.** 당신이 실제로 무엇이 필요한지 묻고, 원본에 철저히 충실하며, *독립적인 비평가*가 렌더링된 슬라이드를 검토하기 전까지는 덱을 "완성"이라고 부르길 거부합니다. 당신이 돌려받는 것은 직접 소유하는, 편집 가능한 진짜 PowerPoint 파일입니다 — 스크린샷도 아니고, 당신을 가둬두는 웹 앱도 아닙니다.

모든 결정을 관통하는 하나의 믿음이 있습니다. **슬라이드는 읽히기 위한 문서가 아니라 발표자를 위한 시각 보조 도구다** — 그래서 모든 것이 *몇 초 안에 이해되는 것*에 최적화됩니다.

---

## 무엇이 다른가

조용하지만 분명한 세 가지 원칙이 이 도구를 흔한 슬라이드 제작 방식과 구분합니다.

- **만들기 전에 인터뷰합니다.** 목적, 청중, 원본, 스타일, 언어 — 처음에 모두 파악하며, 결코 짐작하지 않습니다. 엉뚱한 질문에 자신만만하게 답하는 덱은 이제 없습니다.
- **당신의 작업을 꾸며낼 수 없습니다.** 모든 수치, 주장, 그림은 반드시 당신의 원본까지 추적되어야 합니다. 단 하나의 예외인 전망성 내용은 모델 자신의 추가분으로 명시 표시됩니다. 전문가 청중은 지어낸 결과를 즉시 알아채므로, 애초에 지어내지 않습니다.
- **자기 픽셀을 직접 점검합니다 — 게다가 또 다른 눈으로.** `python-pptx`는 눈을 감고 씁니다. 넘침, 대비, 글리프 버그는 렌더링되고 나서야 드러납니다. 그래서 모든 덱은 이미지로 렌더링되고, **독립적인 비평가 서브에이전트가 동의해야만** 인계됩니다. 빌더가 자기 숙제를 스스로 채점하는 일은 허용되지 않습니다.

### slide-maker 대 흔한 슬라이드 제작 방식

<sub>✓ 예&nbsp;&nbsp;·&nbsp;&nbsp;~ 부분적 / 상황에 따라&nbsp;&nbsp;·&nbsp;&nbsp;✗ 아니오</sub>

| 당신이 얻는 것 | One-shot AI prompt | Web slide tools | By hand (PPT / `python-pptx`) | **slide-maker** |
|---|:--:|:--:|:--:|:--:|
| 만들기 *전에* 목표와 청중을 물어본다 | ✗ | ~ | ✓ | **✓** |
| 원본에 충실하다 — 지어낸 수치가 없다 | ~ | ~ | ✓ | **✓** |
| 원본 자체의 그림을 쓴다 — PDF에서 자동으로 잘라내며, 다시 그리지 않는다 | ✗ | ✗ | ~ | **✓** |
| 독립적인 비평가가 **렌더링된** 슬라이드를 점검한다 | ✗ | ✗ | ✗ | **✓** |
| *목적*에 맞춘 디자인 (디펜스 ≠ 피치 ≠ 강의) | ~ | ~ | ✓ | **✓** |
| 직접 소유하는, 편집 가능한 진짜 `.pptx` — 종속 없음 | ~ | ~ | ✓ | **✓** |
| 어떤 언어든 — CJK 및 진짜 수식 타이포그래피 포함 | ~ | ~ | ✓ | **✓** |
| 재현 가능한 빌드 + 안전한 재편집 | ✗ | ~ | ✓ | **✓** |
| *세련된* 덱까지 빠르게 | ~ | ✓ | ✗ | **✓** |

다른 방식들도 모두 슬라이드는 만들 수 있습니다. slide-maker는 **묻고, 충실하며, 결과를 점검하는** 유일한 도구입니다 — 그러면서도 당신이 온전히 소유하는 파일을 건네줍니다.

## 작동 방식 — 절제된 하나의 루프

> **인터뷰 → 이해 → 빌드 → 렌더 & 비평 ⟲ → 인계**

모든 덱은 일곱 단계를 거칩니다 (`SKILL.md`가 권위 있는 명세입니다).

| 단계 | 무슨 일이 일어나는가 | 왜 존재하는가 |
|---|---|---|
| **0 — 인터뷰** | 하나의 `AskUserQuestion` 묶음: 템플릿, 목적과 청중, 원본 자료, 스타일. (+후속 질문: 학회 장소, 새 템플릿.) | 사용자의 요구사항이 진실의 원천입니다. 이전 덱에서 물려받는 게 아니라 직접 *배웁니다*. |
| **1 — 이해와 기획** | 디스패치된 **콘텐츠 기획 에이전트**가 모든 원본을 깊이 읽고(원본이 없으면 웹에서 조사·사실 확인하고) **이해 브리프**를 작성한 뒤 덱을 설계합니다 — 이 단계와 3단계를 한 명의 머리로 수행하는 한 번의 깊은 작업으로 묶습니다. | 보기엔 멀쩡하지만 작업을 오독한 덱은 어떤 전문가도 속이지 못합니다. 충실함은 여기서 시작됩니다. |
| **2 — 캔버스** | 출력 폴더(`~/Downloads/<deck>/`)를 정하고, 템플릿을 불러오거나 *목적에 맞는* 룩을 디자인합니다. 팔레트/폰트(CJK `EAFONT` 포함)를 설정합니다. | 브랜딩은 레이아웃에 담깁니다. 디자인은 단어를 읽기도 전에 *문서의 종류*를 알려야 합니다. |
| **3 — 기획** | 슬라이드별 사양(핵심 우선: 내용, 시각 소스, 레이아웃, 모션 + 이미지 옵트인), 한 장에 한 아이디어, 분당 ~1장, 목적에 맞춘 흐름. ~15장 이상 → 섹션 팬아웃. **빌드 전에 계획을 보여주고 승인을 받습니다.** | 계획은 고치기 싸지만 완성된 덱은 고치기 비쌉니다. |
| **4 — 빌드** | `deckkit` 헬퍼를 쓰는 하나의 빌드 스크립트. 원본 그림 전체, 등폭 분할 패널(`columns`), 여백, 순환하는 강조색, 진짜 수식, 한 가지 언어, 빌드/애니메이션과 이미지는 **취향과 목적**에 따라(강조/몰입/안내 — 할당량 없음), 발표자 노트. | python-pptx는 빠릅니다. 한 번의 스크립트 실행, 하나의 일관된 저자. |
| **5 — 렌더 + 비평 루프** | PNG로 렌더링해 직접 *봅니다*. 그다음 **독립적인 비평가 서브에이전트**가 JSON을 반환합니다 (동의 / 수정 + 슬라이드별 수정안). 동의할 때까지 반복합니다. | python-pptx는 눈을 감고 씁니다 — 넘침/대비/글리프 버그는 픽셀에서만 보입니다. 당신은 자기 작업의 심판이 될 수 없습니다. |
| **6 — 인계 + 반복** | 사용자에게 보여주고, 폴더 경로를 알려주며, 편집 가능성과 두 가지 변경 경로를 설명하고, 피드백을 반영합니다. | 덱은 사용자의 것이며 계속 — 안전하게 — 다듬을 수 있습니다. |

**액터-비평가 루프가 품질 엔진입니다.** 그 *무게*는 사안의 중요도에 맞게 조절됩니다 — 랩 미팅이면 비평가 한 명, 학회·디펜스·피치라면 서로 다른 관점을 가진 2–3명의 비평가 패널 — 하지만 루프 자체는 절대 생략되지 않습니다.

### 두 가지 모드

- **자동 (기본값):** 인터뷰 → 빌드 → 높은 기준까지 비평가 루프 → 제시. 비평가가 *품질*을 책임집니다.
- **협업 (선택):** 저렴한 **승인 게이트**를 추가합니다 — 실제 렌더링된 옵션 중에서 *방향*을 고르고 → *개요*를 승인한 뒤 → 나머지를 빌드합니다. 게이트는 비평가가 읽어낼 수 없는 *선호*(취향)를 포착합니다. 백지에서 디자인할 때는 **뚜렷이 구별되는 3가지 방향** — 그리고 *"직접 설명하기"* 옵션 — 을 보여주고, 그중 하나를 고른 다음에야 확정합니다.

## 할 수 있는 것

- **무엇으로든 — 또는 아무것도 없이 빌드합니다.** 논문, 코드베이스, 문서, 기존 슬라이드 → 덱. 자료가 없다면? 전문성으로 초안을 잡고 **웹 검색으로 모든 주장을 근거 짓고 사실 확인합니다.**
- **당신의 진짜 그림을, 정확하게 씁니다.** 원본 자체의 그림을 **논문/PDF에서 곧장** 끌어옵니다 — 캡션으로 자동 감지하고 그림의 실제 범위에 맞춰 잘라내(범례와 축은 그대로) 다시 그리거나 잘라먹지 않고 *통째로* 보여줍니다. 빽빽한 비교 그리드는 중요한 열만 골라 재구성할 수 있고, 의심스러운 크롭은 확인하도록 표시됩니다.
- **기존 덱을 다시 디자인합니다.** 먼저 진단하고, 범위를 확인한 뒤, 당신의 내용과 그림을 재사용해 다시 빌드합니다 — 말없이 처음부터 갈아엎는 일은 결코 없습니다.
- **마음에 드는 룩을 맞춥니다.** 예시 하나를 건네면 그 *스타일* — 그리드, 팔레트, 타이포그래피, 모티프 — 을 자체 빌드로 재현합니다.
- **청중의 언어로 말합니다.** 어떤 언어든 전체에 걸쳐 일관되게 유지하며, 제대로 된 **CJK 타이포그래피**와 진짜 **LaTeX 품질의 수식**을 씁니다.
- **장소를 존중합니다.** 학회 발표라면 빌드 전에 장소를 파악하고 조사합니다 — 형식, 종횡비, 공식 템플릿, 청중.
- **큰 덱으로 확장합니다.** 15장 이상 → 공유 스타일, 병렬 저작, 비평가 패널을 갖춘 선택적 섹션 팬아웃.
- **깔끔하게 인계합니다.** 자체 완결형 폴더, 발표자 노트, 목적에 맞는 애니메이션, 그리고 안전하게 계속 편집할 수 있는 재현 가능한 빌드 스크립트.

## 시작하기

slide-maker는 **Agent Skill**입니다 — Claude Code와 그 밖의 Agent-Skills 호환 런타임에서 동작합니다. 사용하려고 명령을 실행할 필요가 없습니다. 그저 **요청하면** 스킬이 알아서 (인터뷰부터 시작해) 넘겨받습니다.

```bash
# 1. 설치 (Claude Code 경로 예시; 모든 Agent-Skills 런타임에서 동작)
git clone https://github.com/dong845/slides_maker ~/.claude/skills/slide-maker

# 2. 일회성 툴체인 점검 (python-pptx, LibreOffice, matplotlib, …)
bash ~/.claude/skills/slide-maker/scripts/check_env.sh
```

그런 다음 에이전트에게 이렇게 요청하기만 하면 됩니다.

> *"paper.pdf로 12분짜리 학회 발표를 만들어줘."*
> *"내 덱이 너무 빽빽해 — 다시 디자인해줘."*
> *"확산 모델에 대한 강의를, 中文으로 — 깔끔하고 다이어그램 위주로."*
> *"이 저장소를 투자자용 피치로 바꿔줘."*

완성된 덱은 `~/Downloads/<deck-name>/`에 도착합니다 — `.pptx`, 슬라이드 PNG가 담긴 `render/`, 그리고 그것을 만든 빌드 스크립트.

## 당신의 요청이 어느 경로를 타는가

인터뷰(0단계, 특히 Q3)가 요청의 경로를 정합니다.

| 사용자가 원하는 것… | 경로 |
|---|---|
| 코드/논문/문서로 만든 덱 | 빌드 경로(1–6단계), 콘텐츠 분기 |
| 자료 없는 덱 | 빌드 경로; 전문성으로 초안 + 웹 검색으로 근거 짓고, 개요 확인 |
| **자신의 덱을 개선**하기 | **재디자인 경로** — 먼저 진단, 범위 확인, 내용/그림 재사용해 재빌드 (`references/redesign-existing-deck.md`) |
| **예시처럼 보이는** 덱 | 스타일 모방 — 스타일 브리프 작성, 룩 재현 (`references/style-analysis.md`) |
| **학회** 발표 | 장소 파악 + 웹 조사(규칙, 템플릿, 청중) 후 그에 맞춰 빌드 |
| **포스터** | 한정 범위: 하나의 큰 캔버스; 제작 원칙은 그대로지만 스킬은 발표에 맞춰져 있음 — 먼저 명세 확인 |
| **비영어 / CJK** 덱 | `EAFONT` 설정, 단일 언어 원칙, CJK 타이포그래피 (`references/multilingual.md`) |
| **큰** 덱 (15장 이상) | 선택적 섹션 팬아웃: 공유 `style.py`, 병렬 섹션 저자, `assemble.py`, 비평가 패널 (`references/large-deck-orchestration.md`) |
| **먼저 옵션을 보기** | 협업 모드 게이트 |
| **납품 후 변경** | 안전하게 반복 — 손수 편집한 내용을 절대 덮어쓰지 않음 (`references/handoff-and-iteration.md`) |

## 스킬에 새겨진 디자인 원칙

1. **산출물보다 요구사항.** 템플릿, 오래된 덱, 모델의 취향은 *입력*이지 지시가 아닙니다. 명시된 요구사항과 충돌할 때는 요구사항이 이깁니다.
2. **엄격한 충실성.** 모든 주장/수치/그림은 원본까지 추적됩니다. 단 하나의 예외는 명확히 표시된 전망성 내용입니다.
3. **독립적인 비평.** 별개의 에이전트가 렌더링된 픽셀을 심판합니다 — 그 독립성이야말로 "동의"에 의미를 부여합니다.
4. **수집은 병렬화하되, 이해는 결코 그러지 않는다.** 읽기/자산 준비는 펼쳐서 진행하되, 하나의 머리가 전체 맥락을 붙듭니다.
5. **목적에 맞춘 디자인.** 디펜스, 임원 보고, 강의가 똑같이 생겨서는 안 됩니다.
6. **하나의 언어, 전체에 걸쳐 일관되게.**
7. **스크립트가 진실의 원천이고, `.pptx`는 산출물이다.** 재현 가능하며, 사용자의 편집을 잃지 않고 안전하게 반복할 수 있습니다.

## 툴체인

`python-pptx`, `pymupdf`(렌더 + 그림 추출), `matplotlib` + `Pillow`(수식/차트/그림 크롭), 그리고 렌더링을 위한 LibreOffice(`soffice`). 새 컴퓨터에서 `bash scripts/check_env.sh`를 한 번 실행하세요. 누락된 것이 있으면 정확한 해결책을 출력해 줍니다.

<details>
<summary><b>저장소 지도</b> (기여자용)</summary>

**중추**
- `SKILL.md` — 모델이 따르는 운영 지침 (0–6단계, 규칙).

**엔진 (`scripts/`)**
- `deckkit.py` — 빌드 키트: 텍스트/도형/컴포넌트 헬퍼(`bullet`, `callout`, `chip`, `arrow`, `modbox`, `hrule`), 레이아웃/이미지 헬퍼(등폭 분할 패널용 `columns`, `picture`), 수식(`eq_par`, `equation_png`), `speaker_notes`, 대비 점검, 팔레트/폰트(CJK `EAFONT` 포함), 템플릿 재사용(`open_template`, `content_slide`), 그리고 템플릿 없는 크롬(`blank_deck`, `title_bar`, `footer`). 가져다 쓰세요 — 기본 요소를 다시 만들지 마세요.
- `render_deck.sh` — `.pptx` → 슬라이드당 PNG 하나 (LibreOffice → PDF → PNG). 크로스 플랫폼; 전용 LibreOffice 프로파일을 사용해 병렬/공존 렌더가 충돌하지 않습니다.
- `check_env.sh` — 툴체인을 위한 일회성 사전 점검.
- `anim.py` — python-pptx가 쓸 수 없는 PowerPoint 빌드/애니메이션 타이밍 XML을 주입합니다.
- `assemble.py` — 병렬 저작된 섹션 모듈을 하나의 덱으로 합칩니다 (취약한 병합 없이).
- `archetypes.py` — 협업 게이트를 위해 방향별로 동일한 미리보기 슬라이드를 빌드합니다.
- `inspect_template.py` — 템플릿의 레이아웃/플레이스홀더/로고를 출력합니다.
- `extract_pdf.py` — 원본 PDF에서 그림을 정확히 자동 감지하고 잘라냅니다 (`figures`/`figure`/`autofig`; 캡션 기준 + 콘텐츠에 스냅, 유효성 점검 포함), 더불어 수동 페이지/영역/임베디드 이미지 추출도 지원합니다.
- `crop_helper.py` — 짐작이 아니라 보면서 이미지를 다룹니다: `grid`(눈금자 오버레이), `crop`/`--snap`, `trim`(콘텐츠에 스냅; 밝거나 어두운 배경에서 범례/축을 잘라먹지 않고 배경을 제거), `panel`(빽빽한 비교 그리드에서 고른 열/행을 재구성).
- `extract_deck.py` — 기존 덱에서 텍스트/표/그림을 *꺼냅니다* (재디자인 + 대조).
- `export_notes.py` — 덱의 발표자 노트를 평문 리허설 스크립트로 내보냅니다.

**판단**
- `agents/content-planner.md` — 건설적 기획자의 브리프: 자료를 깊이 이해하고(또는 웹 조사), 서사 흐름과 슬라이드별 계획(내용, 레이아웃, 모션, 목적에 맞춘 스타일의 이미지)을 설계합니다.
- `agents/critic.md` — 독립 비평가의 브리프 + JSON 스키마.
- `references/review-rubrics.md` — 보편 루브릭 + 목적별 오버레이 (연구 근거 기반).
- `references/design-principles.md` — 기술적 노하우와 그 "이유."

**시나리오별 레퍼런스**
- `design-by-purpose.md` · `animation.md` · `multilingual.md` · `font-guidance.md` · `style-analysis.md` · `redesign-existing-deck.md` · `collaborative-mode.md` · `large-deck-orchestration.md` · `handoff-and-iteration.md`
- `examples/` — 완성된 빌드 스크립트, 공유 스타일 + 섹션 모듈 컨벤션.

**외부 (스킬의 일부 아님)**
- `~/.claude/slide-templates/` — 사용자의 개인 템플릿 레지스트리; 선택을 위해 읽고, 새 프로파일을 여기에 씁니다. 신규 사용자에게는 비어 있습니다.

</details>
