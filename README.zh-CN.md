# slide-maker

> **构建、重做并审校演示级 `.pptx` 幻灯片** —— 面向任何受众、任何语言，无论你是否带着模板或素材。

大多数 AI 工具做幻灯片的方式和它们写文字一样：凭一次猜测一气呵成，却从不真正*看一眼*自己做出了什么。**slide-maker 不同，它的工作方式更像一位资深的演示设计师。** 它会先问清你到底需要什么，严格忠于你的素材，并且在一位*独立评审*审过实际渲染出的幻灯片之前，绝不把一份幻灯片称作“完成”。你最终拿到的，是一份真正属于你、可自由编辑的 PowerPoint 文件 —— 不是一张截图，也不是一个把你锁死在里面的网页应用。

有一个信念贯穿每一个决策：**幻灯片是讲者的视觉辅助，而不是供人阅读的文档** —— 所以一切都为*几秒之内被看懂*而优化。

---

## 它为何与众不同

三条低调的纪律，把它和惯常的做幻灯片方式区分开来：

- **先访谈，再动手。** 目的、受众、素材、风格、语言 —— 这些都在动手前问清楚，从不靠假设。再也不会有那种自信满满、却答错了问题的幻灯片。
- **它无法编造你的工作。** 每一个数字、每一项主张、每一张图，都必须可追溯回你的素材；唯一的例外 —— 前瞻性内容 —— 会被明确标注为模型自己的补充。专家受众一眼就能识破一个杜撰的结果，所以它干脆不杜撰。
- **它检查自己的每一个像素 —— 还请来第二双眼睛。** `python-pptx` 是“盲写”的：溢出、对比度、字形错误只有在渲染之后才会暴露。所以每一份幻灯片都会被渲染成图像，并且必须有一位**独立的评审子代理点头同意**才能交付。构建者无权给自己的作业打分。

## 它如何运作 —— 一个严守纪律的闭环

> **访谈 → 理解 → 构建 → 渲染并评审 ⟲ → 交付**

每一份幻灯片都流经七个步骤（`SKILL.md` 是权威规范）：

| 步骤 | 发生了什么 | 它为何存在 |
|---|---|---|
| **0 — 访谈** | 一批 `AskUserQuestion` 提问：模板、目的与受众、素材、风格。（外加追问：会议场合、新模板。） | 用户的需求才是事实之源；你要*学到*它们，绝不从此前的幻灯片里继承。 |
| **1 — 理解与规划** | 由调度的**内容规划 agent**深读全部素材（无素材时联网检索并核实），写出一份**理解摘要**，再设计整套幻灯片——本步骤与步骤 3 合为一次由同一头脑完成的深度推进。 | 一份看着对、却误读了工作的幻灯片骗不过任何专家。忠实，从这里开始。 |
| **2 — 画布** | 决定输出文件夹（`~/Downloads/<deck>/`），载入模板*或*设计一套贴合目的的外观；设定配色/字体（含中日韩 `EAFONT`）。 | 品牌信息存在版式之中；设计应当在第一个字被读到之前，就传递出这是*哪一类*文档。 |
| **3 — 规划** | 逐页规格（结论先行：内容、视觉来源、版式、动效 + 配图选择项），每页一个想法，页数约 1 页/分钟，叙事弧线契合目的；约 15+ 页 → 分节展开。**构建前先展示计划供你确认。** | 改计划很便宜；改一份做完的幻灯片很贵。 |
| **4 — 构建** | 一个构建脚本，使用 `deckkit` 助手函数。整张原图、等宽分栏（`columns`）、留白沟槽、轮换的点缀色、真正的公式、单一语言、逐步呈现/动画与配图依**品味与目的**（强调／吸引／引导——无配额）、讲者备注。 | python-pptx 很快；一次脚本运行，一位连贯一致的作者。 |
| **5 — 渲染 + 评审闭环** | 渲染成 PNG 并*亲眼看*；随后一位**独立评审子代理**返回 JSON（同意 / 修改 + 逐页修复项）。循环直至同意。 | python-pptx 是盲写的 —— 溢出/对比度/字形错误只在像素里现形。你不是自己作品的裁判。 |
| **6 — 交付 + 迭代** | 展示给用户、给出文件夹路径、说明可编辑性与两条改动通道、纳入反馈。 | 这份幻灯片归用户所有，可以一直安全地继续打磨。 |

**这个“演员–评审”闭环就是质量引擎。** 它的*分量*随风险等级伸缩 —— 实验室例会一位评审就够，会议、答辩或路演则上一个 2–3 人、视角各异的评审小组 —— 但闭环本身从不跳过。

### 两种模式

- **自动（默认）：** 访谈 → 构建 → 评审闭环冲到高标准 → 展示。评审把关的是*质量*。
- **协作（可选开启）：** 加入低成本的**审批关卡** —— 从实际渲染出的选项里挑一个*方向* → 批准*大纲* → 再构建其余部分。关卡把关的是*偏好*（品味），这是评审读不出来的。从零开始设计时，它会先给你**3 个截然不同的方向** —— 外加一个*“描述你自己的想法”* —— 让你在它动手之前先做选择。

## 它能做什么

- **从任何东西构建 —— 或者什么都没有也行。** 一篇论文、一个代码库、一份文档，或已有的幻灯片 → 一份幻灯片。没有素材？它会凭专业知识起草，并**联网检索来佐证并核实**每一项主张。
- **精准地用上你真正的图。** 它会**直接从论文/PDF 里**拉取素材自带的图 —— 按图注自动识别，并按图的真实范围裁切（图例与坐标轴完好无损），*完整*呈现，而不是重画或截断。密集的对比网格可以重新拼合成只保留要紧的那几列；可疑的裁切会被标记出来供你过目。
- **重做你现有的幻灯片。** 它先诊断，确认范围，再复用你的内容与图表来重建 —— 绝不悄悄地推倒重来。
- **复刻你喜欢的外观。** 给它一个范例，它就能在自己的构建里重现那种*风格* —— 栅格、配色、字体、母题。
- **说你受众的语言。** 任意语言，全程保持一致，配以恰当的**中日韩排版**与真正的 **LaTeX 级公式**。
- **尊重会场。** 对于会议演讲，它会在动手前识别并研究会场 —— 格式、宽高比、官方模板、受众。
- **撑得起大型幻灯片。** 15+ 页 → 可选的分节展开，共享统一风格、并行撰写，并配一个评审小组。
- **干净利落地交付。** 一个自包含的文件夹、讲者备注、有意图的动画，以及一个可复现的构建脚本，让你能安全地继续编辑。

## 试用它

slide-maker 是一个 **Agent Skill** —— 它运行在 Claude Code 以及其他兼容 Agent Skills 的运行时里。你不用敲命令去用它；你只需**开口提要求**，技能就会接管（从访谈开始）。

```bash
# 1. 安装（这里演示 Claude Code 路径；任何 Agent Skills 运行时都可用）
git clone https://github.com/dong845/slides_maker ~/.claude/skills/slide-maker

# 2. 一次性的工具链检查（python-pptx、LibreOffice、matplotlib 等）
bash ~/.claude/skills/slide-maker/scripts/check_env.sh
```

然后只管对你的 agent 说：

> *“用 paper.pdf 做一个 12 分钟的会议演讲。”*
> *“我的幻灯片太密了 —— 帮我重做。”*
> *“一节关于扩散模型的课，用 中文 —— 干净、多图。”*
> *“把这个代码仓库变成一份投资人路演。”*

做好的幻灯片会落在 `~/Downloads/<deck-name>/` —— 包括那份 `.pptx`、一个装着各页 PNG 的 `render/`，以及生成它的构建脚本。

## 你的请求会走哪条路径

访谈（步骤 0，尤其是第 3 问）会为请求选路：

| 用户想要…… | 路径 |
|---|---|
| 用自己的代码/论文/文档做一份幻灯片 | 构建路径（步骤 1–6），内容分支 |
| 一份没有素材的幻灯片 | 构建路径；凭专业知识起草 + 联网检索来佐证，确认大纲 |
| **改进自己现有的**幻灯片 | **重做路径** —— 先诊断、确认范围，复用其内容/图表来重建（`references/redesign-existing-deck.md`） |
| 一份**看起来像某个范例的**幻灯片 | 风格仿制 —— 写一份风格摘要，重现那种外观（`references/style-analysis.md`） |
| 一场**会议**演讲 | 识别并联网研究会场（规则、模板、受众），再据此构建 |
| 一张**海报** | 限定范围：单张大画布；制作准则依然成立，但本技能是为演讲调优的 —— 先确认规格 |
| 一份**非英语 / 中日韩**幻灯片 | 设置 `EAFONT`、单一语言纪律、中日韩排版（`references/multilingual.md`） |
| 一份**大型**幻灯片（15+ 页） | 可选的分节展开：共享 `style.py`、并行的分节作者、`assemble.py`、评审小组（`references/large-deck-orchestration.md`） |
| **先看看选项** | 协作模式关卡 |
| **交付后的改动** | 安全迭代 —— 绝不覆盖手工编辑（`references/handoff-and-iteration.md`） |

## 烙进技能里的设计原则

1. **需求高于产物。** 一个模板、一份旧幻灯片，或模型自己的品味，都是*输入*，而非指令。当它们与既定需求冲突时，需求胜出。
2. **严格忠实。** 每一项主张/数字/图表都可追溯回素材。唯一的例外是被清楚标注的前瞻性内容。
3. **独立评审。** 由一个单独的代理来评判渲染出的像素 —— 正是它的独立性，让“同意”二字有了分量。
4. **并行化收集，绝不并行化理解。** 把阅读/素材准备铺开去做；但要有一个头脑统揽贯穿全篇的主线。
5. **设计贴合目的。** 一场答辩、一份高管汇报、一节讲课，不该长得一个样。
6. **单一语言，全程贯穿。**
7. **脚本才是事实之源；`.pptx` 只是产物。** 可复现，且能在不丢失用户编辑的前提下安全迭代。

## 工具链

`python-pptx`、`pymupdf`（渲染 + 图片提取）、`matplotlib` + `Pillow`（公式/图表/图片裁切），以及用于渲染的 LibreOffice（`soffice`）。在新机器上跑一次 `bash scripts/check_env.sh`；任何缺失项，它都会打印出确切的修复办法。

<summary><b>仓库地图</b>（面向贡献者）</summary>

**主干**
- `SKILL.md` —— 模型遵循的操作说明（步骤 0–6 与各项规则）。

**引擎（`scripts/`）**
- `deckkit.py` —— 构建套件：文本/形状/组件助手（`bullet`、`callout`、`chip`、`arrow`、`modbox`、`hrule`），版面/图片助手（`columns`/`rows` 用于等宽分栏与堆叠面板、`picture`），公式（`eq_par`、`equation_png`），`speaker_notes`，对比度检查，配色/字体（含中日韩 `EAFONT`），模板复用（`open_template`、`content_slide`）以及无模板的版面装饰（`blank_deck`、`title_bar`、`footer`）。直接导入它；别再去重造这些基本件。
- `render_deck.sh` —— `.pptx` → 每页一张 PNG（LibreOffice → PDF → PNG）。跨平台；使用一个私有的 LibreOffice 配置文件，使并行/共存的渲染互不冲突。
- `check_env.sh` —— 工具链的一次性预检。
- `anim.py` —— 注入 python-pptx 写不了的 PowerPoint 逐步呈现/动画计时 XML。
- `assemble.py` —— 把并行撰写的分节模块合成一份幻灯片（不靠脆弱的合并）。
- `archetypes.py` —— 为协作关卡按每个方向构建同一批预览页。
- `inspect_template.py` —— 打印一个模板的版式/占位符/标识。
- `extract_pdf.py` —— 把一张图从源 PDF 里*抽取*出来：`figures`/`figure`/`autofig` **从论文里自动识别并精准裁切图**（以图注为锚 + 贴合内容，附带有效性检查），外加手动的页面/区域/内嵌图片提取。

- `extract_deck.py` —— 把文本/表格/图表从一份现有幻灯片里*抽取*出来（重做 + 对账）。
