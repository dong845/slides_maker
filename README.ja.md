# slide-maker

> **プレゼンテーション品質の `.pptx` デッキを、構築・再設計・批評する** — あらゆる聴衆に、あらゆる言語で、テンプレートや元素材の有無を問わず。

たいていの AI ツールは、テキストを書くのと同じやり方でスライドを作ります。一発勝負で、当て推量から、しかも自分が何を生み出したかを一度も*見る*ことなく。**slide-maker は、その代わりにベテランのプレゼンデザイナーのように働きます。** 本当に必要なものは何かを問いかけ、あなたの元素材に厳格に忠実であり続け、そして*独立した批評者*がレンダリング済みのスライドをレビューするまで、デッキを「完成」と呼ぶことを拒みます。手元に返ってくるのは、あなた自身が所有する本物の編集可能な PowerPoint ファイルです。スクリーンショットでもなければ、囲い込まれた Web アプリでもありません。

ひとつの信念が、あらゆる判断を貫いています。それは、**スライドは話者のための視覚的補助であって、読ませる文書ではない**ということ。だからこそ、すべては「数秒で理解できること」に向けて最適化されます。

---

## 何が違うのか

通常のスライド作成法とは一線を画す、3つの静かな規律があります。

- **構築する前に、まずインタビューする。** 目的、聴衆、元素材、スタイル、言語 — これらを最初にすべて聞き取り、決して前提で済ませません。間違った問いに自信たっぷりに答えてしまうデッキは、もう生まれません。
- **あなたの仕事を捏造できない。** あらゆる数値、主張、図は、必ずあなたの元素材まで遡れなければなりません。唯一の例外である将来志向のコンテンツは、モデル自身の追加として明示されます。専門家の聴衆は、でっち上げた結果を一瞬で見抜きます。だからこそ、でっち上げないのです。
- **自分のピクセルを、もう一組の目とともにチェックする。** `python-pptx` は目隠しで書き出します。オーバーフロー、コントラスト、グリフのバグは、レンダリングして初めて姿を現します。だからこそ、すべてのデッキは画像へレンダリングされ、**独立した批評サブエージェントの同意が必須**となってから引き渡されます。作り手が、自分の答案を自分で採点することはできません。

|---|:--:|:--:|:--:|:--:|
| 構築する*前に*目的と聴衆を尋ねる | ✗ | ~ | ✓ | **✓** |
| 元素材に忠実 — 数値をでっち上げない | ~ | ~ | ✓ | **✓** |
| 元素材自身の図を使う — PDF から自動でクロップし、描き直さない | ✗ | ✗ | ~ | **✓** |
| 独立した批評者が**レンダリング済み**スライドをチェック | ✗ | ✗ | ✗ | **✓** |
| *目的*に合わせた設計（学位審査 ≠ ピッチ ≠ 講義） | ~ | ~ | ✓ | **✓** |
| 自分が所有する本物の編集可能な `.pptx` — 囲い込みなし | ~ | ~ | ✓ | **✓** |
| あらゆる言語 — CJK や本物の数式組版も含む | ~ | ~ | ✓ | **✓** |

どのツールもスライドは作れます。けれども slide-maker は、**問いかけ、忠実であり続け、結果をチェックする**唯一のツールであり — しかもなお、あなたが完全に所有するファイルを手渡します。

## 仕組み — 規律ある一本のループ

> **インタビュー → 理解 → ビルド → レンダー＆批評 ⟲ → 引き渡し**

すべてのデッキは7つのステップを通って流れていきます（正式な仕様は `SKILL.md`）。

| ステップ | 何が起きるか | なぜ存在するか |
|---|---|---|
| **0 — インタビュー** | 1回の `AskUserQuestion` バッチで、テンプレート、目的と聴衆、元素材、スタイルを確認（＋フォローアップ：カンファレンス会場、新規テンプレート）。 | ユーザーの要件こそが真実の源。要件は*学び取る*ものであり、過去のデッキから引き継ぐものではありません。 |
| **1 — 理解とプラン** | 派遣された**コンテンツプランナー・エージェント**がすべての元素材を深く読み込み（素材がなければウェブで調査・事実確認し）、**理解ブリーフ**を作成したうえでデッキを設計します——本ステップとステップ3を一人の頭脳による一度の深い工程としてまとめます。 | 見た目は正しくても内容を読み違えたデッキは、いかなる専門家も欺けません。忠実性はここから始まります。 |
| **2 — キャンバス** | 出力フォルダ（`~/Downloads/<deck>/`）を決定し、テンプレートを読み込むか、目的に合った見た目を設計する。パレットとフォント（CJK の `EAFONT` を含む）を設定。 | ブランディングはレイアウトに宿ります。設計は、一語も読まれる前に、その文書が*どういう種類のもの*かを伝えるべきです。 |
| **3 — プラン** | スライドごとの仕様（テイクアウェイ優先：内容、ビジュアル出典、レイアウト、モーション＋画像のオプトイン）、各スライド1アイデア、約1枚/分、目的に合わせた弧。約15枚以上ならセクションのファンアウトへ。**ビルド前にプランを提示して承認を得ます。** | プランの修正は安く、完成したデッキの修正は高くつきます。 |
| **4 — ビルド** | `deckkit` ヘルパーを使った1本のビルドスクリプト。元の図をまるごと活用し、等幅の分割パネル（`columns`）、ガター、ローテーションするアクセント、本物の数式、単一言語、ビルド/アニメーションと画像は**センスと目的**に従って（強調／引き込み／誘導——ノルマなし）、スピーカーノートを備える。 | python-pptx は高速。1回のスクリプト実行で、一貫した一人の作者によるデッキになります。 |
| **5 — レンダー＋批評ループ** | PNG にレンダリングして*見る*。次に**独立した批評サブエージェント**が JSON を返す（同意 / 修正＋スライドごとの修正点）。同意が得られるまでループ。 | python-pptx は目隠しで書き出します。オーバーフロー・コントラスト・グリフのバグはピクセルでしか現れません。あなたは自分自身の仕事の審判にはなれないのです。 |
| **6 — 引き渡し＋反復** | ユーザーに見せ、フォルダパスを伝え、編集可能性と2つの変更レーンを説明し、フィードバックを取り込む。 | デッキはユーザー自身のものであり、安全に調整し続けられます。 |

**アクター・批評ループこそが品質エンジンです。** その*重み*は重要度に応じてスケールしますが（ラボミーティングなら批評者1名、カンファレンス・学位審査・ピッチなら異なる観点を持つ2〜3名の批評パネル）、ループそのものが省かれることは決してありません。

### 2つのモード

- **自動（デフォルト）：** インタビュー → ビルド → 高い基準に達するまでの批評ループ → 提示。批評者が*品質*を担保します。
- **協働（オプトイン）：** 安価な**承認ゲート**を追加します。実際にレンダリングされた選択肢から*方向性*を選び → *アウトライン*を承認 → 残りをビルド。ゲートは、批評者には読み取れない*嗜好*（テイスト）を捉えます。ゼロから設計する場合は、コミットする前に選べる**3つの異なる方向性** — さらに*「自分で記述する」*という選択肢も加えて — を提示します。

## できること

- **何からでも — あるいは何もなくても構築する。** 論文、コードベース、ドキュメント、既存スライド → デッキへ。素材がない場合は、専門知識から下書きし、あらゆる主張を**裏付け・ファクトチェックするために Web 検索を行います**。
- **あなたの本物の図を、正確に使う。** 元素材自身の図を**論文/PDF から直接**取り出します — キャプションで自動検出し、図の本来の範囲どおりに（凡例や軸を保ったまま）クロップして、描き直したり切り刻んだりせず*まるごと*表示します。詰め込まれた比較グリッドは、重要な列だけに再構成でき、怪しいクロップは確認のためにフラグが立てられます。
- **既存のデッキを再設計する。** まず診断し、スコープを確認してから、あなたのコンテンツと図を再利用して再構築します。無言のゼロからの作り直しは決してしません。
- **気に入った見た目に合わせる。** 例を渡せば、その*スタイル* — グリッド、パレット、タイポグラフィ、モチーフ — を、自前のビルドの中で再現します。
- **聴衆の言語で語る。** あらゆる言語に対応し、全編を通じて一貫して保持。適切な **CJK タイポグラフィ**と本物の **LaTeX 品質の数式**を備えます。
- **会場に敬意を払う。** カンファレンス講演なら、構築の前に会場を特定して調査します — フォーマット、アスペクト比、公式テンプレート、聴衆まで。
- **大規模デッキにもスケールする。** 15枚以上なら、共有スタイルによる任意のセクションファンアウト、並列の著作、そして批評パネルへ。
- **きれいに引き渡す。** 自己完結したフォルダ、スピーカーノート、目的に沿ったアニメーション、そして安全に編集を続けられるよう再現可能なビルドスクリプトを添えて。

## 試してみる

slide-maker は **Agent Skill** です — Claude Code や、その他の Agent-Skills 対応ランタイムで動作します。使うのにコマンドを実行する必要はありません。ただ**頼むだけ**で、スキルが（インタビューから）引き継いでくれます。

```bash
# 1. インストール（Claude Code の場合を例示。Agent-Skills 対応ランタイムなら何でも動作）
git clone https://github.com/dong845/slides_maker ~/.claude/skills/slide-maker

# 2. 一度きりのツールチェーン確認（python-pptx、LibreOffice、matplotlib など）
bash ~/.claude/skills/slide-maker/scripts/check_env.sh
```

あとは、エージェントにこう頼むだけです。

> *「paper.pdf から12分のカンファレンス講演を作って。」*
> *「私のデッキは詰め込みすぎ — 再設計して。」*
> *「拡散モデルの講義を、中文で — クリーンで図表中心に。」*
> *「このリポジトリを投資家向けピッチに変えて。」*

完成したデッキは `~/Downloads/<deck-name>/` に届きます — `.pptx`、スライド PNG の `render/`、そしてそれを作ったビルドスクリプトとともに。

## あなたのリクエストがたどる経路

インタビュー（ステップ0、特に Q3）がリクエストを振り分けます。

| ユーザーが望むこと… | 経路 |
|---|---|
| 自分のコード/論文/ドキュメントからのデッキ | ビルド経路（ステップ1〜6）、コンテンツ分岐 |
| 素材なしのデッキ | ビルド経路。専門知識から下書きし、Web 検索で裏付け、アウトラインを確認 |
| **自分のデッキを改善**したい | **再設計経路** — まず診断し、スコープを確認し、ユーザーのコンテンツ/図を再利用して再構築（`references/redesign-existing-deck.md`） |
| **例のような見た目**のデッキ | スタイル模倣 — スタイルブリーフを書き、その見た目を再現（`references/style-analysis.md`） |
| **カンファレンス**講演 | 会場を特定して Web 調査し（ルール、テンプレート、聴衆）、それに合わせてビルド |
| **ポスター** | 限定対応：単一の大型キャンバス。制作の原則は通用するが、本スキルは講演向けにチューニングされているため、まず仕様を確認 |
| **非英語 / CJK** のデッキ | `EAFONT` を設定し、単一言語の規律、CJK タイポグラフィ（`references/multilingual.md`） |
| **大規模**デッキ（15枚以上） | 任意のセクションファンアウト：共有 `style.py`、並列セクション著者、`assemble.py`、批評パネル（`references/large-deck-orchestration.md`） |
| **まず選択肢を見たい** | 協働モードのゲート |
| **引き渡し後の変更** | 安全に反復 — 手作業の編集を決して上書きしない（`references/handoff-and-iteration.md`） |

## スキルに組み込まれた設計原則

1. **アーティファクトよりも要件。** テンプレート、古いデッキ、モデルのテイストはいずれも*入力*であって指示ではありません。それらが明示された要件と衝突したときは、要件が勝ちます。
2. **厳格な忠実性。** あらゆる主張/数値/図は元素材にまで遡れること。唯一の例外は、明確に表示された将来志向のコンテンツです。
3. **独立した批評。** 別のエージェントがレンダリングされたピクセルを審査します。その独立性こそが「同意」を意味あるものにします。
4. **収集は並列化し、理解は決して並列化しない。** 読み込みと素材準備はファンアウトし、一筋の論旨は一人の頭が保持します。
5. **目的に合った設計。** 学位審査、経営層への報告、講義が同じ見た目であってはなりません。
6. **単一言語を、最後まで貫く。**
7. **スクリプトが真実の源であり、`.pptx` はアーティファクトである。** 再現可能で、ユーザーの編集を失うことなく安全に反復できます。

## ツールチェーン

`python-pptx`、`pymupdf`（レンダリング＋図の抽出）、`matplotlib` ＋ `Pillow`（数式/チャート/図のクロップ）、そしてレンダリング用の LibreOffice（`soffice`）。新しいマシンでは一度 `bash scripts/check_env.sh` を実行してください。不足しているものに対する正確な対処法を出力します。
